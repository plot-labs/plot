# 0001. Monorepo With Generated Apps

## Status

Accepted

## Decision

Plot uses a monorepo:

- `apps/web` is generated by `create-next-app`.
- `apps/api` is reserved for the backend service once the production runtime is
  selected.
- Shared TypeScript packages live under `packages`.
- Product, architecture, API, and operations notes live under `docs`.

The backend stays as one service for v0. The first autonomous loop is still
backend-managed: watch a selected shipped-work source through the first adapter,
import shipped changes, load Writing Blocks, a Content Template, and a Voice
Profile; call the model provider; then persist the Agent Run, Content Pack,
Content Variants, citations, and style metadata. A separate agent
service, direct publishing, policy checks, and broader automation can be added
later without changing the monorepo shape.

## Rationale

This keeps the product small enough to build while still making the backend
architecture visible for portfolio review. The frontend scaffold avoids
hand-rolled setup, while the backend domain modules make Plot's v0 loop
explicit:

```txt
shipping window -> source adapter -> Writing Blocks -> Content Template -> Voice Profile -> Agent Run -> API Generation -> Content Pack -> citations/style guidance
```

## Consequences

- Frontend and backend can evolve independently but ship from one repo.
- The API is the source of truth for domain state, repository watches, agent runs, generation runs, prompt inputs, and generated variants.
- v0 includes a lightweight update-agent loop but does not require a separate agent runtime, content checks, handoff exports, or publish automation.
- Generation can start as a synchronous or simple job-backed model-provider call inside that update-agent loop.
- The frontend should consume a generated API client once OpenAPI is wired.
- Landing pages start inside `apps/web`; `apps/landing` can be split later if SEO or marketing operations justify it.
